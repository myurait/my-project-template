# Ideal Experience Interview Template

Use this template when the project lacks a valid ideal experience spec or when the existing one is no longer trustworthy.

- Date:
- Interviewer:

- Related docs:

## 1. User Context

- Who is the primary user?
- What environment do they work in?
- What devices do they switch between?
- What kind of projects and interruptions are normal?

## 2. Current Pain

- What is frustrating today?
- What gets repeated unnecessarily?
- What breaks the flow of work?
- What feels like tool management instead of actual work?

## 3. Ideal Experience

- If the product worked perfectly, what would the user say they can now do naturally?
- What should feel automatic?
- What should the user no longer have to choose, explain, or organize manually?
- What should still remain under explicit user control?

## 4. Conversation and Context Continuity

- How should a new conversation relate to an old one?
- What should happen when the system guesses the wrong context?
- What kind of correction flow feels natural?
- What prior knowledge should be remembered automatically?

## 5. Project Elevation

- How does vague wall-bouncing become a project or structured work item?
- What friction must disappear before the user feels this is working?
- What document or planning work should the system absorb on behalf of the user?

## 6. Worker Orchestration

- What should the front agent decide automatically?
- What should remain overridable?
- What must be visible about downstream workers?
- What should never spill raw into the main chat?

## 7. Success and Failure Signals

- How will we know we built the intended experience?
- What user behaviors or example conversations would count as success?
- What outcomes would indicate we solved the wrong problem?
- What tradeoffs would be unacceptable even if the implementation is technically correct?
